# Linker golden corpus

10 hand-verified document pairs with ground-truth `(item_a, item_b, link_kind)`
expectations. Each pair is engineered to exercise one specific link kind so the
gate measures the agent on the precise dimension being tested.

Per `docs/superpowers/specs/2026-05-19-internal-knowledge-linking-design.md` § 6
and `attune-pro/docs/agent-skill-training-methodology.md` § 2 "cardinal rule:
independent derivation" — ground truth here is hand-derived (regex applied
manually + visual diff against the source text), **not** generated by calling
the agent and asserting its own output back.

Each fixture YAML carries:

- `id` — fixture identifier (filename prefix)
- `kind` — one of `shared_entity` / `semantic_near` / `explicit_ref`
- `doc_a.title` / `doc_a.content` / `doc_a.url` — the citing / first item
- `doc_b.title` / `doc_b.content` / `doc_b.url` — the cited / second item
- `expected.link_kinds` — link kinds the agent must emit for the pair
- `expected.evidence_substring` — substring that must appear in the emitted
  link's `evidence` field (e.g. `"person:张三"` or `"url:https://..."`)

Source corpora ground truth is derived from:

- Tech corpus (Rust ownership / borrow / lifetimes / async): paraphrased from
  the upstream rust-lang/book chapters (MIT/Apache-2.0). Document pairs
  contrived to share specific entities or explicit URL refs.
- Operating-systems / DB corpus (cs-notes): same approach.
- Domain mixing: technology vs operations write-ups so semantic similarity is
  observable.

These are **engineered fixtures** — small documents that target a single
hypothesis. The golden gate `linker_golden_gate.rs` loads all `*.yaml` files in
this directory and runs every pair through `compute_links_for_item`,
asserting the expected link kind appears in `list_links_for_item` and the
evidence substring matches.

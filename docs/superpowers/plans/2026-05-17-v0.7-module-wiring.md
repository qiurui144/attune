# v0.7 模块接线 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development（推荐，模块独立、可并行）或 superpowers:executing-plans 逐模块实现。步骤用 checkbox(`- [ ]`)跟踪。
>
> **本文档是 master 计划** —— 11 个模块是 11 个独立子系统。每个模块执行前应由执行 agent
> 跑一遍 writing-plans 产出该模块的 bite-sized 详细计划（本 master 给定接线面 + OSS 方案 + 验收）。

**Goal:** 把 `71d82ee` 落地但孤儿化的 11 个 v0.7 P0 模块接线成可用功能 —— 6 个真逻辑模块接路由/引擎/UI，5 个 scaffold 模块用成熟开源方案补真实现。

**Architecture:** attune-core 提供模块逻辑 → attune-server `routes/` 暴露 HTTP → `ui/src/` 暴露界面。scaffold 模块各自的 `trait + MockXxxProvider` 已就位，OSS 实现 = 在同 trait 下加 `RealXxxProvider`，AppState 按配置选 Mock/Real。

**Tech Stack:** Rust(attune-core/server)· Preact+Vite(ui)· OSS crate:`imap` / `frankenstein` / `reqwest_dav` · 前端 `pdf.js` · VLM 复用现有 `LlmProvider` vision。

---

## 原则（per 用户决策 2026-05-17）

1. **少自研、优先开源** —— scaffold 模块的真实现尽量是"集成 1 个成熟 crate"，不手写协议。
2. **不破坏现有 trait** —— `EmailProvider`/`TelegramProvider`/`VlmProvider`/`BackupProvider` 等已定义；OSS 实现挂在同 trait 下，Mock 保留给测试。
3. **每模块走 CLAUDE.md 代码变更流程** —— 2 轮 review + 测试 + README/DEVELOP/RELEASE 同步。

## 共享文件协调（多模块都改 → 必须串行或单 agent 收口）

以下文件几乎每个模块都碰，**禁止多 agent 并行编辑**，由收口任务统一处理：
- `attune-core/src/lib.rs` —— 模块已 `pub mod`，无需改
- `attune-server/src/routes/mod.rs` —— 每个新路由模块加一行 `pub mod`
- `attune-server/src/lib.rs` —— 每个新路由加 `.route(...)`
- `attune-server/src/state.rs` —— scaffold 模块的 provider 字段 + accessor
- `ui/src/i18n/{zh,en}.ts` —— 新 UI 文案
- `rust/RELEASE.md`

**协调策略**：并行 agent 各自在 worktree 改自己模块的*新文件*；上述共享文件的改动每个 agent 写成 patch 片段交收口任务串行 apply。或：串行执行（更稳，慢）。

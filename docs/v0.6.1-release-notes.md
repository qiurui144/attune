# Attune v0.6.1 — Release Notes

_Released: 2026-04-30_

## Headline

Attune v0.6.1 is a **boundary + form-factor patch**. We tightened the OSS × Pro story across all README/spec docs, gave Attune a `FormFactor` axis so K3 appliance images can default to local Ollama while laptops keep the v0.6.0 GA "remote-token-default" behavior, and patched 3 RUSTSEC vulnerabilities in `rustls-webpki`. **No breaking change** — v0.6.0 laptop users see identical default behavior.

## What's new

- **`FormFactor` axis** (`platform/mod.rs`): new `Laptop` / `K3Appliance` / `Server` / `Unknown` enum on `HardwareProfile`. Detection order: `ATTUNE_FORM_FACTOR` env var override > Linux DMI keyword (`K3` / `Jetson`) > default `Laptop`.
- **`default_settings` form-factor split** (`routes/settings.rs`): `Laptop` / `Server` / `Unknown` keep the v0.6.0 default `llm.provider = "openai_compat"` (remote token, wizard guides API key entry); `K3Appliance` defaults to `provider: "ollama"` + `endpoint: "http://localhost:11434/v1"` + `model: "qwen2.5:3b"` for K3 image with pre-installed local LLM.
- **`/status/diagnostics` exposes `form_factor` + `prefers_local_llm`** so the desktop wizard can show the right default.
- **Wizard Step 3 (LLM) form-factor-aware** (`Step3LLM.tsx`): K3 hosts get an Ollama-first card layout with ★ Recommended; laptop hosts get a Cloud-first layout. The non-recommended card stays available with a dashed border.
- **`PREFERRED_MODELS` doc-comment clarified** (`llm.rs`): explicitly states this is the Ollama auto-detect fallback list, **not** the default behavior. Triggered only when the user picks Ollama mode in the wizard or `settings.llm.provider = "ollama"`.

## Documentation convergence (OSS × Pro boundary)

Six documentation drift points were resolved to bring code, CLAUDE.md, public README, and the 4-17 positioning spec back to a single truth:

- README (`/README.md` + `.zh.md` + `rust/README.md` + `rust/README.zh.md`): top tagline + "Who it's for" table now reflect "any individual knowledge worker" with a separate **Industry users** section pointing at `attune-pro/<vertical>-pro` plugin packs.
- `docs/oss-pro-strategy.md` (and `.zh.md`): §3 Decision 3 Lite tier no longer claims "all 4 builtin basic plugins"; §4.1 OSS scope adds Persona system row (4 generic personas, 🟡 v0.7+); §4.2 Pro scope adds Industry personas row (Lawyer / Doctor / PatentAgent etc.).
- `docs/superpowers/specs/2026-04-17-product-positioning-design.md`: top banner notes that §5 user/scenario sections have been migrated to `attune-pro/docs/specs/<vertical>-pro-vertical-workflow-design.md`; §3 LLM wording updated to reflect form-factor split.
- `CLAUDE.md`: new "Git 分支管理标准（GitFlow Lite）" section documents two long-lived branches, develop→main triggers, tag dual-track, `--no-ff` requirement, and the `--first-parent` invariant for "main has no non-merge commits".
- `attune-pro/docs/specs/`: new `law-pro` / `patent-pro` / `academic-pro` vertical workflow design specs (508 lines total), each mapping the L1-L5 value stack from the OSS-Pro strategy.

## Security

- `rustls-webpki` 0.103.10 → 0.103.13 — fixes 3 CVEs:
  - RUSTSEC-2026-0099 (name constraints accepted for wildcard names)
  - RUSTSEC-2026-0098 (name constraints for URI names incorrectly accepted)
  - RUSTSEC-2026-0104 (reachable panic in CRL parsing)
- 0 remaining vulnerabilities; 8 unmaintained-dep warnings remain (core2 / fxhash / instant / paste / rustls-pemfile / lru — non-security).

## Tests

- 622 tests passed (lib + integration + binary), 0 failed, 3 ignored. Up from "237+" reported in v0.6.0 GA — the count includes all tests in `attune-core` lib (535) + `attune-server` lib (5) + `attune-server-headless` (3) + 16 integration suites.
- 4 new `FormFactor` unit tests in `platform/mod.rs::tests` cover env-var override / default / `prefers_local_llm` / `HardwareProfile::detect` integration.
- 4 new `default_settings` unit tests in `routes/settings.rs::tests` verify Laptop → remote token, K3 → local Ollama, Server/Unknown → fallback to remote token, non-LLM settings invariant across form factors.

## Compatibility

- v0.6.0 laptop users: **zero behavior change**. `default_settings` for Laptop/Server/Unknown produces identical output to v0.6.0 (remote token default).
- K3 appliance image builders: set `ATTUNE_FORM_FACTOR=k3` in the systemd service unit (or rely on DMI keyword detection), and the wizard will default to local Ollama with pre-installed `qwen2.5:3b`.
- No database schema change. No vault migration needed.

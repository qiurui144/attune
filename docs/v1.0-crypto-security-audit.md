## 2. 维度 1: AES-GCM 字段加密

### 2.1 版本确认

`aes-gcm = "0.10"` — 当前主版本，无已知 CVE。

### 2.2 Nonce 生成

**结论: ✅ 正确实现**

`rust/crates/attune-core/src/crypto.rs` 中 `encrypt()` 实现：

```rust
let mut nonce_bytes = [0u8; NONCE_LEN]; // NONCE_LEN = 12
OsRng.fill_bytes(&mut nonce_bytes);
let nonce = Nonce::from_slice(&nonce_bytes);
```

- **每次调用均生成独立随机 nonce**，无全局状态，无复用风险
- 使用 `OsRng`（系统熵源），注释中明确说明"不用 thread_rng (非密码学保证)"
- Nonce 与密文拼接存储 `nonce || ciphertext`，解密时 `split_at(NONCE_LEN)` 正确分离

### 2.3 Key Derivation (Argon2id)

**结论: ✅ 参数合理**

```rust
pub const ARGON2_M_COST: u32 = 65536; // 64 MB
pub const ARGON2_T_COST: u32 = 3;
pub const ARGON2_P_COST: u32 = 4;
```

与 OWASP 推荐最低值（m=19456/64MB, t=2/3, p=1/4）对齐，满足攻击成本要求。

**Salt 生成**: `generate_salt()` 使用 OsRng，32 字节随机盐，每次初始化独立生成。

**Input 拼接**: `password || device_secret` 组合派生，防止仅凭密码字典攻击还原主密钥。输入数据使用 `Zeroizing<Vec<u8>>` 保护。

### 2.4 加密字段范围

通过 vault.rs 追踪加密路径：
- DEK (Data Encryption Key) 本身被 MK 加密存储: `encrypted_dek_db`, `encrypted_dek_idx`, `encrypted_dek_vec`
- DEK 备份亦加密: `encrypted_dek_*_recovery`
- 用户数据内容通过 DEK 加密（items.content、annotations 等，走 `crypto::encrypt(dek, plaintext)` 路径）

### 2.5 内存安全

- `Key32` 实现 `Zeroize + ZeroizeOnDrop`，Drop 时自动清零
- `derive_master_key` 的 input buffer 使用 `Zeroizing<Vec<u8>>`

### 2.6 不足之处

无高危问题。轻微改善建议（v1.1）：
- AES-GCM 最大可加密 ~64 GB（2^39 字节），attune 场景无法触达，但可加注释说明边界

## 3. 维度 2: Vault Session Token

### 3.1 Token 结构

**结论: ✅ 设计合理**

Token 格式: `{session_id}:{expires}:{nonce}`

- `session_id`: UUID v4 simple（`uuid::Uuid::new_v4()`），128-bit 随机
- `expires`: UTC timestamp，TTL = 4 小时 (`SESSION_TTL_SECS = 4 * 3600`)
- `nonce`: 单调递增计数器，vault 级别撤销凭证
- 签名: HMAC-SHA256，使用 master_key 签名 payload

整个 token = `payload:hex(hmac_sign(mk, payload))`

### 3.2 验证流程

`verify_session()` 流程：

1. 分离 payload 与 sig
2. `hmac_verify()` 调用 `mac.verify_slice(signature).is_ok()` — **常量时间比较**（由 `hmac` crate 的 `CtMac::verify_slice` 保证）
3. 解析并验证 `expires > now`
4. 验证 `token_nonce == current_nonce`（防止 lock 操作后旧 token 复用）

### 3.3 撤销机制

- `lock()` 操作：`increment_token_nonce()` — 所有旧 token 立即失效
- `change_password()` 操作：同样 increment nonce（OSS-S6 修复）
- 不存在"无限期有效 token"漏洞

### 3.4 Bearer Token 传输

中间件从 `Authorization: Bearer <token>` header 提取，**不从 URL query string 读取 token**。日志中对 query string 的 `token` 参数做了 `redact_query_param()` 脱敏处理。

### 3.5 ⚠️ 待跟进: `no_auth` flag 未被 `cfg(test)` 隔离

```rust
// headless.rs
no_auth: bool,  // CLI 参数，未限制 cfg(test)
```

`no_auth = true` 时所有非 `ALWAYS_AUTH_ENDPOINTS` 端点绕过 Bearer auth。`lib.rs` 仅打印 `warn!`，不阻止启动。

**风险评估**: 中低。`no_auth` 仅对 loopback 地址有效（非 loopback 有 warn 但可继续），默认值为 `false`，生产部署文档需明确禁用此 flag。

## 4. 维度 3: SQL Injection

### 4.1 整体状况

**结论: ✅ 通过，无高危注入**

对 `attune-core/src/store/` 下所有 `.rs` 文件扫描，几乎全部使用：
- `prepare_cached(static_sql)` + `?` placeholder
- `prepare(&sql)` + `params_from_iter()`（动态但参数化）

### 4.2 动态 SQL 案例分析

**案例 1**: `store/signals.rs` — `count_citation_hits_for_refs()`

```rust
let placeholders = std::iter::repeat("?").take(refs.len()).collect::<Vec<_>>().join(",");
let sql = format!("SELECT COUNT(*) FROM skill_signals WHERE kind = 'citation_hit' AND ref_id IN ({placeholders})");
let mut stmt = self.conn.prepare(&sql)?;
stmt.query_row(rusqlite::params_from_iter(refs.iter()), |r| r.get(0))
```

占位符数量动态，但**值全部走参数绑定**，SQL 结构字符串中不含用户输入。✅ 安全。

**案例 2**: `store/mod.rs` — `wipe_all_user_data()`

```rust
let safe = table.replace('"', "\"\"");
let sql = format!("DELETE FROM \"{safe}\"");
```

表名来自 `table_names` 常量列表（内部定义），不是用户输入。对双引号做转义只是防御性编码。✅ 安全（⚠️ 建议将 `table_names` 的来源注释更明确）。

## 5. 维度 4: 越权（Paid Feature Guard）

### 5.1 当前状态

**结论: ⚠️ 待补充**

`member_session::MemberState` 区分 `LoggedOut / Free / Paid`，并通过 `login-token` 和 `login-password` 端点设置。`is_paid()` 方法存在。

但审计发现：
- 缺乏统一的 **route-level paid feature guard middleware**
- `chat.rs`、`llm.rs`、`settings.rs` 等路由中未发现 `is_paid()` 检查
- 付费特性（如 tiered assembler、LLM provider 配置等）的访问控制**仅在客户端逻辑区分**

### 5.2 风险评估

**中级风险**。当前 attune OSS 定位是本地工具，`member_state` 主要用于 UI 显示和云端 gateway 配额追踪，不是安全边界。但 v1.0 若对外声称"pro 特性需付费"，服务端无 guard 则任何知道 API 的用户可直接访问付费功能。

### 5.3 建议

v1.0 前需确认：
1. 列出明确的"仅 paid 可用" API 端点清单
2. 为这些端点添加 `if !state.member_state.lock()...is_paid() { return Forbidden }` guard
3. 或通过 middleware layer 覆盖付费路由组

## 6. 维度 5: 文件上传 Sanitize

### 6.1 文件大小限制

**结论: ✅ 双道防线**

- 框架层: `DefaultBodyLimit::max(100 * 1024 * 1024)` — 100 MB
- 应用层: `MAX_UPLOAD_BYTES = 100 * 1024 * 1024` 二次检查

两处注释明确说明需同步修改，设计意图清晰。

### 6.2 文件类型过滤

**结论: ✅ Parser 层拒绝可执行文件**

`attune-core/src/parser.rs` 对已知二进制格式有明确拒绝：

```rust
// 已知二进制格式（video/archive/executable 等）拒绝，避免乱码入库
"unsupported file format '{ext}': only text, code, documents, spreadsheets, images and audio are accepted"
```

允许扩展名包括 `.py`, `.js`, `.ts`, `.rs`, `.sh`, `.bash` 等代码文件（视为文本内容），但这些文件只会被解析为**纯文本**存入知识库，不会被执行。

### 6.3 路径遍历防护

**结论: ✅ 多层防护**

`routes/index.rs` — `validate_bind_path()` 三重检查：
1. `is_absolute()` — 拒绝相对路径
2. `dunce::canonicalize()` — 解析符号链接
3. `canonical.starts_with(home)` — 限制在用户 home 目录内

### 6.4 ⚠️ 待改善: Magic Bytes 校验缺失

当前仅依赖文件扩展名判断类型，不做 magic bytes（文件头）校验。用户可将 `.exe` 重命名为 `.txt` 上传，parser 会将其作为文本解析（可能产生乱码内容，但不会执行）。

**风险评估**: 低。attune 是本地工具，用户上传自己的文件，无远程代码执行路径。但 v1.1 可添加 `infer`/`magic` crate 做内容验证。

## 8. v1.0 Ship 风险评估

### 8.1 结论

**v1.0 可以 ship，需在 5/25 GA 前解决以下 2 项阻塞项：**

| 优先级 | 项目 | 阻塞 v1.0? |
|--------|------|-----------|
| P0 | cargo audit 在 CI 跑通并确认无 CVE | ✅ 阻塞 |
| P0 | 确认或补充 paid feature server-side guard（或明确文档说明"本地工具无需服务端付费强制"） | ✅ 阻塞（取决于产品定位） |
| P1 | 添加基础 deny.toml | 建议但不阻塞 |
| P2 | no_auth flag 加 `#[cfg(debug_assertions)]` 保护或文档强调禁止生产使用 | 建议 |
| P3 | Magic bytes 校验（upload） | v1.1 |
| P3 | rand 0.8 → 0.9 升级 | v1.1 |

### 8.2 整体安全状态

加密核心（AES-GCM + Argon2id + HMAC-SHA256 + Zeroize）实现**质量高于市场同类本地 AI 工具平均水平**。主要风险集中在运营/配置层面（audit CI、no_auth flag），而非密码学实现层面。

## 9. 高风险 Finding 列表

### F-1: cargo audit 无法执行（P0 阻塞）

- **位置**: CI 配置 / 环境
- **描述**: SSH key 缺失导致 advisory DB 无法更新，无法确认第三方依赖无已知 CVE
- **修复**: CI pipeline 加 `cargo audit --db <https-url>` 或配置 HTTPS 替代 SSH

### F-2: Paid Feature 无 Server-Side Guard（P0 — 产品决策依赖）

- **位置**: `attune-server/src/routes/` 各付费功能路由
- **描述**: `MemberState::Paid` 存在但 chat/LLM 等路由未见 `is_paid()` 检查
- **修复选项 A**: 添加 paid route middleware（如 v1.0 付费特性有明确 API 边界）
- **修复选项 B**: 明确文档说明"attune OSS 是本地单用户工具，付费特性由 cloud gateway 控制配额，server 端无需强制"

### F-3: `no_auth` 可在生产 CLI 使用（中级）

- **位置**: `attune-server/src/bin/headless.rs:21`
- **描述**: `--no-auth` 是 CLI 参数，无 `#[cfg(debug_assertions)]` 或 feature flag 隔离，生产用户可能误用
- **修复**: 加 `warn!` 提示或加 `#[cfg(debug_assertions)]` 限制
